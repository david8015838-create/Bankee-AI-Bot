🧠 SOP 助理系統運作邏輯

完整說明此 AI 助理如何理解你的問題並找到答案

📌 一句話總結
這個助理會把你的 SOP 文件「拆成小段」進行語意理解，當你提問時，先精準找出答案藏在哪幾份文件，再把整份文件交給 AI 閱讀後回答。所有運算都在你的瀏覽器內進行，沒有後端伺服器。
瀏覽器直接運算 語意搜尋 小段搜尋｜整份注入 勾選驅動索引 三道護欄

1 整體架構

這個工具是一個靜態網頁，沒有任何後端伺服器。當你打開頁面，瀏覽器會直接跟以下三方服務溝通：

  • Google Drive：抓取部門 SOP 資料夾中的文件內容
  • Gemini 語意搜尋服務：把文件和你的提問轉換成「語意指紋」用於比對
  • Gemini 對話服務：根據找到的文件內容生成回答

你的問題、文件、API Key 等資料完全留在你的瀏覽器與這三方服務之間，沒有經過任何「中介伺服器」。

2 文件如何進入系統

  1. 列出資料夾：點「開啟文件庫」後，系統去 Google Drive 抓取指定資料夾裡的所有檔案清單。
  2. 下載內容：依檔案類型用對應方式取得內文（Google 文件會自動轉成純文字）。
  3. 本機快取 24 小時：下載過的文件存在你的瀏覽器，24 小時內不會重複下載。
  4. 背景建索引：載入完成後，系統會自動為你勾選的文件建立語意索引（見下一段）。第一次使用預設全部勾選，之後會記住你的偏好。

3 勾選驅動索引（重要）

系統採用「勾選 = 建立索引」的設計，讓你決定哪些文件納入搜尋範圍：

  • 勾選一份文件 → 系統在背景幫它建立語意索引（變成綠色 ⚡）
  • 取消勾選 → 立刻刪掉這份的索引，釋放瀏覽器儲存空間
  • 全選 / 取消全選 → 批次處理上面兩個動作
  • 狀態會記住：下次打開頁面，勾選狀態保留不變
為什麼這樣設計？部門總共可能有幾十份 SOP，但每個人通常只關心自己負責的領域。讓你只勾選相關的文件，可以節省瀏覽器儲存空間，搜尋也會更精準（不會被無關文件干擾）。

4 文件切分策略

每份文件會被切成多個小段（chunk），切分規則：

  • 切點：按文件裡的標題切（一級標題、二級標題、三級標題）。例如「適用對象」「前置準備」「步驟 1」「步驟 2」各成一段。
  • 每段帶有標題資訊：系統知道每一小段屬於文件的哪個章節。
  • 過短才捨棄：少於 3 個字才會被視為空段落丟掉（過去設 30 字，會誤殺「印表機 IP：192.168.1.100」這類短條目，已修正）。
  • 沒有標題的文件：整份視為一個大段落（退化處理）。
為什麼按標題切，而不是按固定字數？

SOP 文件的「適用對象 / 前置準備 / 步驟 / FAQ」這些段落本身就是完整的語意單位。按標題切剛好對齊語意邊界。固定字數會把同一個步驟切兩半，導致電腦無法理解。

單段上限：每段送進語意搜尋前會截到前 3000 字。但這只影響「某一個標題下面內容超過 3000 字」的極端情境，一般 SOP 段落幾百字而已，遠遠不會撞到。

5 語意指紋（Embedding）配置

所謂「語意指紋」就是把一段文字轉換成 768 個數字組成的陣列。意思相近的文字會有相近的指紋，這就是語意搜尋的基礎。

主要使用模型gemini-embedding-001（Google 提供，目前免費）
備援模型gemini-embedding-002（當主模型暫時不能用時自動切換）
指紋大小用途標籤— 這是 Google 推薦的最佳配對，對中文搜尋準確度有明顯提升
單段字數限制速度控制 6 提問到答案的完整流程
  1. 問題轉指紋：你輸入問題後，系統把問題送給 Gemini 算出問題的「語意指紋」。
  2. 挑選候選池：從你勾選的文件中，取出所有小段的指紋當候選。
  3. 計算相似度：拿問題指紋跟每個小段比對，給出 0 到 1 之間的相似度分數。
  4. 每份文件取最高分：每份文件中相似度最高的那一段代表整份文件的分數。
  5. 套用三道護欄：分數 ≥ 0.45 → 依分數排名 → 最多取前 5 份。
  6. 組合注入內容：先放釘選的常駐文件，再放命中的文件，總字數不超過 60,000 字。
  7. 請 AI 回答：把上述內容當參考資料交給對話模型，AI 根據文件回答並標示來源。
💡 為什麼用小段搜尋、卻把整份文件交給 AI？

小段提供精準定位能力（能在「工作基本資訊」這種雜項型文件中找出其中一條），但 AI 需要文件的完整上下文才能正確回答（例如知道步驟 3 卻不知道步驟 1、2 是什麼）。所以「用小段找、給整份讀」是兩全其美。

7 📌 常駐文件

某些「不管問什麼都可能需要」的參照型文件（部門通訊錄、系統路徑速查表、職稱對照表等），內容雜項太多，語意搜尋不一定能穩定命中。這類文件可以釘選為常駐。

  • 運作方式：釘選後，每次提問都會直接放進參考資料，不需要被搜尋命中。
  • 優先順序：常駐文件永遠排在語意搜尋結果之前注入。
  • 字數限制：仍受 60,000 字總上限約束。常駐文件加總若已超標，後面的搜尋結果可能會被截掉。

什麼時候會自動清理 / 更新？

  • 取消勾選文件 → 立刻刪掉這份文件的所有指紋
  • 編輯文件後存檔 → 系統發現內容變了，自動重新建指紋
  • 換 embedding 模型 → 舊指紋視為失效（不同模型的指紋無法互相比對），自動重建
  • 從 Drive 刪除的文件 → 下次載入時系統會發現並清掉殘留指紋
  • 儲存空間滿了 → 在側邊狀態列顯示警告，提示你取消勾選釋放空間
  • 點「🔄 重新整理文件」 → 全部清除重建

10 語意搜尋失效時的備援

以下狀況系統會自動退回「直接使用所有勾選文件」的舊行為，確保服務不中斷：

  • 主要 AI 不是 Gemini 時（語意搜尋目前只接 Gemini）
  • 沒有設定 API Key
  • 還沒勾選任何文件 / 還沒建索引
  • 沒有任何文件相似度超過 0.45
  • Gemini 語意服務暫時故障

每次回答上方會以小標籤顯示本次用的模式：🟢 語意搜尋 或 🟡 全文模式，點擊可以看到每份文件的相似度分數。

12 資料流向與隱私邊界

🌐 會離開瀏覽器送出的內容
  • 送到 Google Drive：登入授權 + 文件下載/上傳請求
  • 送到 Gemini 語意搜尋服務：你勾選的文件內容（建指紋時，每段一次）、你的提問
  • 送到 Gemini 對話服務：命中的文件全文 + 對話歷史 + 你的問題
🔒 永遠不離開瀏覽器
  • 你的 API Keys
  • 個人筆記 / 待辦 / 通訊錄 / 常用連結
  • 語意指紋庫（雖然文件內容被送去算指紋，但指紋本身不會再送給任何服務）
  • 對話歷史（重新整理頁面就清空）

13 使用的模型

  • 語意搜尋主模型：gemini-embedding-001（Google AI Studio，免費額度內可用）
  • 語意搜尋備援：gemini-embedding-002（主模型不可用時自動切換）
  • 對話模型預設：gemini-2.5-flash-lite（可在 ⚙️ API 設定切換成其他 Gemini / OpenAI / Anthropic 模型）
  • 對話備援：當前模型失敗時，會自動換同服務商的其他模型重試；多個 API Key 也會輪流使用

14 已知限制

  • 語意搜尋僅支援 Gemini：把對話模型切到 OpenAI / Anthropic 時，整套搜尋退回備援模式（使用所有勾選文件）。
  • 瀏覽器儲存上限約 5MB：以每份 SOP 約 100-200 KB 估算，安全容量約 25-35 份。撞到時系統會在狀態列警示，取消勾選不相關文件可立即釋放空間。長期超過 40 份建議升級到 IndexedDB（容量 50MB+）。
  • 跨文件依賴：若答案需要綜合多份文件，但問題語意只命中其中一份，可能會有資訊缺漏（解法：把參照型文件設為 📌 常駐）。
  • 單份文件過長：超過 60,000 字會被字數護欄截斷或排除。
  • 沒有使用紀錄：純前端架構無法追蹤誰問了什麼問題（若需要稽核紀錄需要加後端）。
